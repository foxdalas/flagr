# Аналитика и записи данных

С каждым результатом оценки Flagr может делать две вещи — по отдельности или вместе. Обе — это **рекордеры**, включаемые через `FLAGR_RECORDER_ENABLED=true` и выбираемые (через запятую) в `FLAGR_RECORDER_TYPE`:

- **Встроенная аналитика (`datar`)** — ведёт подсчёты в памяти и отдаёт их через два REST-эндпоинта. Никаких внешних зависимостей. Описана первой, ниже.
- **Стриминг (`kafka` / `kinesis` / `pubsub`)** — отправляет каждый результат во внешний конвейер для вашего хранилища или экспериментальной платформы. См. **Стриминговые записи** в конце страницы.

Что бы вы ни использовали, **переключатель `dataRecordsEnabled` на уровне флага управляет записью этого флага во *все* рекордеры** — задайте его в UI (элемент *Data records* в редакторе флага) или через `PUT /api/v1/flags/{id}`. Флаг с выключенной записью никогда не пишется ни в Datar, ни в Kafka, Kinesis или Pub-Sub.

## Встроенная аналитика (Datar)

Datar — это необязательный движок агрегированной аналитики, встроенный в Flagr и работающий в памяти. Он подсчитывает количество оценок в разрезе флага, варианта, сегмента и часа, а затем отдаёт результаты через два REST-эндпоинта — без внешнего конвейера, без Kafka-консьюмера и без отдельного аналитического стека.

## Когда использовать

Простое решение без зависимостей для несложной агрегированной аналитики. Берите его, когда нужны базовые подсчёты оценок с разбивкой по варианту и сегменту, но без поднятия внешнего конвейера.

Prometheus хорошо покрывает метрики на основе rate и временные ряды по вариантам, но он не умеет индексировать по `segment_id` из-за высокой кардинальности. Используйте Datar, когда вам нужно:

- **Разбивка по сегментам** — сколько оценок получил каждый сегмент
- **Исторические итоги** — накопительные подсчёты (а не только rate) за дни или недели
- **Дашборды по флагам** — простая сводка по всем флагам без отдельного аналитического стека

## Включение

Требует `FLAGR_RECORDER_ENABLED=true`. Когда главный переключатель включён, добавьте `datar` в `RecorderType`, чтобы активировать агрегатор в памяти:

```bash
export FLAGR_RECORDER_ENABLED=true
export FLAGR_RECORDER_TYPE=kafka,datar
export FLAGR_RECORDER_DATAR_FLUSH_INTERVAL=60s    # default
```

Таблица `datar_hourly_events` создаётся автоматически через AutoMigrate при запуске. Миграция схемы не нужна.


## Запись
Запись в Datar включается при трёх условиях:

1. `FLAGR_RECORDER_ENABLED=true` (главный переключатель)
2. Наличие `datar` в `FLAGR_RECORDER_TYPE`
3. Переключатель на уровне флага `dataRecordsEnabled: true` (настраивается через `PUT /api/v1/flags/{id}`)


Это значит, что запись можно включать выборочно для отдельных флагов, даже когда Datar включён глобально.

> **Примечание:** после создания или изменения флага подождите хотя бы один цикл обновления кэша оценки (по умолчанию ~3 с), прежде чем отправлять оценки. Кэшу оценки нужно подхватить конфигурацию нового флага, иначе оценки будут возвращать «not found» и не дойдут до рекордера Datar.

## Эндпоинты

### GET /api/v1/datar/summary

Возвращает флаги с агрегированными итогами за временное окно. **В выдачу попадают только те флаги, по которым в этом окне реально был трафик оценок** — флаги без трафика исключаются.

| Параметр | Тип | По умолчанию | Описание |
|-------|------|---------|-------------|
| `from` | RFC 3339 | 7 дней назад | Начало временного окна |
| `to` | RFC 3339 | сейчас | Конец временного окна |
| `limit` | int | 100 | Максимум результатов |
| `offset` | int | 0 | Смещение в результатах |

Ответ:

```json
{
  "flags": [
    {
      "flagID": 1,
      "flagKey": "my-feature",
      "enabled": true,
      "description": "Controls feature X",
      "totalEvalCount": 45283,
      "lastEvaluatedAt": "2026-05-22T14:30:00Z"
    }
  ]
}
```

### GET /api/v1/datar/flags/{flagID}/summary

Детальная разбивка по одному флагу. Возвращает трафик, сгруппированный по варианту, сегменту и дню — все три массива отсортированы (по убыванию счётчика для варианта и сегмента, по возрастанию даты для дня).

| Параметр | Тип | По умолчанию | Описание |
|-------|------|---------|-------------|
| `from` | RFC 3339 | 7 дней назад | Начало временного окна |
| `to` | RFC 3339 | сейчас | Конец временного окна |

Ответ:

```json
{
  "flagID": 1,
  "trafficByVariant": [
    { "variantID": 1, "count": 30188 },
    { "variantID": 2, "count": 15095 }
  ],
  "trafficBySegment": [
    { "segmentID": 10, "count": 30188 },
    { "segmentID": 20, "count": 15095 }
  ],
  "trafficByDay": [
    { "date": "2026-05-21", "count": 22100 },
    { "date": "2026-05-22", "count": 23183 }
  ]
}
```

## Модель данных

Счётчики раскладываются по часовым корзинам через `time.Now().Truncate(time.Hour)`. Каждая строка таблицы `datar_hourly_events` соответствует одной уникальной комбинации:

- `flag_id` — оценённый флаг
- `variant_id` — совпавший вариант
- `segment_id` — совпавший сегмент (0, если ни один сегмент не подошёл)
- `bucket_hour` — усечённая до часа метка времени

Уникальный индекс по `(flag_id, variant_id, segment_id, bucket_hour)` гарантирует, что аддитивные UPSERT-ы работают корректно при одновременной работе нескольких инстансов.

## Потребление ресурсов

- **CPU**: ~87 нс на оценку на горячем пути (существующий ключ), ~98 нс для новых ключей; ноль аллокаций
- **RAM**: ~210 байт на активный кортеж (flag, variant, segment); ~2,1 МБ на 10K ключей
- **Запись в БД**: одна пакетная транзакция раз в интервал сброса (настраивается, по умолчанию 60 с)
- **Рост таблицы**: ~2,4K строк в месяц на 100 флагов (часовые корзины, без очистки)

## Ограничения

- Данные хранятся в памяти до периодического сброса. Если процесс упадёт, может потеряться до одного интервала сброса агрегированных данных (для дашбордовой аналитики это приемлемо).
- Встроенной политики хранения данных нет — таблица растёт без ограничений. При необходимости настройте cron-задачу или политику хранения.
- Подсчёта уникальных сущностей нет (ни HyperLogLog, ни аналогов). Каждая оценка считается один раз вне зависимости от идентичности сущности.

## Стриминговые записи

Когда `FLAGR_RECORDER_TYPE` включает `kafka`, `kinesis` или `pubsub`, Flagr отправляет в этот бэкенд **по одному сообщению на каждую оценку** (и сразу в несколько, если вы перечислили больше одного). Брокеры и учётные данные настраиваются в разделе [Конфигурация сервера → Data Recorder](flagr_env); здесь же описан **формат сообщения**, которое получает ваш консьюмер.

### Конверт фрейма

Каждое сообщение — это `DataRecordFrame`. Его форма зависит от `FLAGR_RECORDER_FRAME_OUTPUT_MODE`:

- **`payload_string`** (по умолчанию) — результат оценки кодируется в JSON-строку, поэтому конверт остаётся стабильным даже при эволюции схемы результата, а полезную нагрузку можно зашифровать:

```json
{ "payload": "{\"flagID\":42,\"variantKey\":\"treatment\", ... }", "encrypted": false }
```

- **`payload_raw_json`** — результат оценки встраивается как вложенный объект, по которому проще делать запросы прямо в хранилище:

```json
{ "payload": { "flagID": 42, "variantKey": "treatment", "...": "..." } }
```

### Полезная нагрузка записи

Полезная нагрузка — это тот же **результат оценки**, который возвращает [API оценки](flagr_eval_api), плюс контекст оценки:

| Поле | Описание |
|-------|-------------|
| `flagID` / `flagKey` | Оценённый флаг |
| `flagSnapshotID` | Использованная ревизия флага — сопоставляется с журналом истории/аудита |
| `segmentID` | Совпавший сегмент (`0`, если ни один) |
| `variantID` / `variantKey` | Назначенный вариант (пусто / `0`, если нет) |
| `variantAttachment` | JSON-вложение варианта |
| `evalContext` | Полный контекст запроса: `entityID`, `entityType`, `entityContext`, … |
| `timestamp` | Когда произошла оценка (UTC) |

### Шифрование (только Kafka)

В режиме `payload_string` рекордер **Kafka** может шифровать полезную нагрузку — задайте `FLAGR_RECORDER_KAFKA_ENCRYPTED=true` и `FLAGR_RECORDER_KAFKA_ENCRYPTION_KEY`. Тогда фрейм несёт base64-шифртекст с `"encrypted": true`; ваш консьюмер расшифровывает его тем же ключом (AES, схема `simplebox`). Шифрование применимо только к режиму `payload_string` — Kinesis и Pub-Sub всегда отправляют `"encrypted": false`.
