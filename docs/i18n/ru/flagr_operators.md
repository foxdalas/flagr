# Операторы условий

**Сегмент** нацеливается на аудиторию с помощью списка **условий**. Каждое условие — это одно правило вида:

```
<property>  <operator>  <value>
```

Все условия в сегменте объединяются по `AND`: сущность подходит сегменту, только если совпадает **каждое** условие. Сегмент без условий подходит всем.

Условия проверяются по `entityContext` сущности — карте «ключ-значение», которую вы передаёте в [запросе на оценку](flagr_eval_api). `property` — это ключ в контексте, который нужно прочитать; `value` — то, с чем его сравнивают.

## 12 операторов

| Оператор | Символ | Совпадает, когда… | Формат значения | Пример |
|----------|--------|---------------|--------------|---------|
| `EQ` | `==` | свойство равно значению | строка в кавычках / число / bool | `"CA"`, `42`, `true` |
| `NEQ` | `!=` | свойство не равно значению | строка в кавычках / число / bool | `"CA"` |
| `LT` | `<` | свойство меньше значения | число | `18` |
| `LTE` | `<=` | свойство ≤ значения | число | `18` |
| `GT` | `>` | свойство больше значения | число | `21` |
| `GTE` | `>=` | свойство ≥ значения | число | `21` |
| `EREG` | `=~` | свойство соответствует регулярному выражению | regex в кавычках | `"^(CA\|NY)$"` |
| `NEREG` | `!~` | свойство не соответствует регулярному выражению | regex в кавычках | `"^test"` |
| `IN` | `IN` | свойство — одно из значений | JSON-массив | `["CA", "NY"]` |
| `NOTIN` | `NOT IN` | свойства нет в списке | JSON-массив | `["CA", "NY"]` |
| `CONTAINS` | `CONTAINS` | строковое свойство содержит подстроку | строка в кавычках | `"premium"` |
| `NOTCONTAINS` | `NOT CONTAINS` | строковое свойство не содержит подстроку | строка в кавычках | `"premium"` |

## Правила кавычек (прочитайте сначала это)

Это самая частая ошибка. Flagr разбирает значение как **выражение**, поэтому его тип важен:

- **Строки нужно брать в двойные кавычки** — `"CA"`, а не `CA`. Слово без кавычек вроде `CA` разбирается как *имя переменной*, а не как литерал, поэтому оно молча не совпадает никогда.
- **Числа пишутся без кавычек** — `18`, а не `"18"`. Число в кавычках — это строка, и она не будет сравниваться через `<`, `>`, `>=` и т. п.
- **Булевы значения пишутся без кавычек** — `true` / `false`.
- **`IN` / `NOTIN` принимают JSON-массив** — `["CA", "NY", "TX"]`.
- **`EREG` / `NEREG` принимают regex в кавычках** — `"^US.*"`.

!> Берите строковые значения в кавычки. `state == "CA"` совпадает; `state == CA` разбирает `CA` как переменную и не совпадает ни с чем — без всякой ошибки. И редактор условий в UI, и утилита `flagr-validate` это подсвечивают, но лучше запомнить это раз и навсегда.

### Форма в UI vs. форма в JSON-файле

Значение всегда хранится как строка. Как его записывать — зависит от места:

- В **UI** вы вводите значение как есть: `"CA"`, `21`, `["US","CA"]`.
- В **JSON-файле флага** значение само по себе является JSON-строкой, поэтому внутренние кавычки экранируются: `"\"CA\""`, `"21"`, `"[\"US\",\"CA\"]"`. См. [JSON-источник флагов](flagr_json_flag_spec).

## Свойства и контекст сущности

`property` — это ключ в `entityContext` запроса на оценку. Например, под такой запрос:

```json
{ "entityID": "u123", "entityContext": { "state": "CA", "age": 31 } }
```

подходит сегмент с такими условиями:

| Свойство | Оператор | Значение |
|----------|----------|-------|
| `state` | `EQ` | `"CA"` |
| `age` | `GTE` | `21` |

Имена свойств могут содержать точки, дефисы и другие символы — Flagr внутри их оборачивает, поэтому `country.code` или `user-tier` работают как ключи свойств.

## Примеры

**Список разрешённых стран** — `country IN ["US","CA","GB"]`:

| Свойство | Оператор | Значение |
|----------|----------|-------|
| `country` | `IN` | `["US", "CA", "GB"]` |

**Числовой порог** — `age >= 21`:

| Свойство | Оператор | Значение |
|----------|----------|-------|
| `age` | `GTE` | `21` |

**Домен почты через regex** — `email =~ "@example\.com$"`:

| Свойство | Оператор | Значение |
|----------|----------|-------|
| `email` | `EREG` | `"@example\\.com$"` |

## Заметки о регулярных выражениях

`EREG` / `NEREG` используют синтаксис `regexp` из Go (RE2). Передавайте шаблон как **строку в кавычках**: `"^v[0-9]+"`.

- Экранирование обратным слешем работает внутри кавычек — `"\\d+"` для цифр, `"\\."` для литеральной точки.
- Шаблоны привязываются только там, где вы их привязали: используйте `^` и `$` для совпадения по всей строке (`"^(CA|NY)$"`), иначе шаблон совпадёт в любом месте значения.
- **Литеральный `/` в шаблоне — единственная шероховатость.** Обычно Flagr внутри оборачивает ваш шаблон в форму regex-литерала (`/…/`) — именно это и позволяет надёжно пропускать экранирование обратным слешем вроде `"\\d+"` и `"\\."`. Шаблон, который *сам* содержит `/`, эту обёртку пропускает и разбирается как обычная строка в кавычках, где экранирование обрабатывает уже парсер выражений и поведение может отличаться. Если нужно сопоставить слеш, проверьте условие в **Консоли отладки** (или через `flagr-validate`), прежде чем на него полагаться.

## Проверка условий

- В **UI** редактор условий показывает встроенную подсказку, когда значение выглядит некорректно (строка без кавычек, нечисловое значение для `<`, некорректный JSON-массив или неверный regex), и держит кнопку Save отключённой, пока ошибка не исправлена.
- Для флагов из **JSON-источника** запустите `flagr-validate` (см. [JSON-источник флагов](flagr_json_flag_spec)) — он проверяет каждое выражение условия перед деплоем.

См. также: [Как работает оценка](flagr_evaluation) — о том, как совпавший сегмент превращается в вариант.
