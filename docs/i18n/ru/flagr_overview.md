# Обзор Flagr

## Основные понятия Flagr

Определения этих понятий есть в [документации API](https://foxdalas.github.io/flagr/api_docs).

- **Флаг (Flag)**. Может быть фича-флагом, экспериментом или конфигурацией.
- **Тег (Tag)**. Поясняющая метка на флаге — упрощает поиск и оценку.
- **Вариант (Variant)** — одно из возможных значений флага. Например, control/treatment или green/yellow/red.
- **Вложение варианта (Variant Attachment)** — динамическая конфигурация варианта. Например, для варианта с кнопкой `green` можно на лету задавать конкретный hex-оттенок зелёного (скажем, `{"hex_color": "#42b983"}`).
- **Сегмент (Segment)** — часть аудитории, на которую мы нацеливаемся. Это наименьшая единица, которую можно анализировать в Flagr Metrics.
- **Условие (Constraint)** — правило, по которому задаётся аудитория сегмента. Иными словами, аудиторию сегмента описывает набор условий. В Flagr условия внутри сегмента объединяются по `AND`.
- **Распределение (Distribution)** — то, как варианты распределяются внутри сегмента.
- **Сущность (Entity)** — то, чему мы назначаем вариант, вместе с её контекстом. Обычно Flagr ждёт, что контекст придёт вместе с сущностью, — тогда по нему можно строить условия.
- **Охват (Rollout)** и детерминированная псевдослучайность. Задача — чтобы результат оценки для каждой сущности был детерминированным и стабильным. Как оценивается флаг по контексту сущности:
    - Берём уникальный ID сущности и считаем от него хеш функцией с равномерным распределением (например, CRC32 или MD5).
    - Берём значение хеша (в десятичной системе) по модулю 1000. Всего в Flagr 1000 «корзин» (buckets).
    - Учитываем распределение. Например, деление 50/50 между control и treatment — это корзины 0–499 под control и 500–999 под treatment.
    - Учитываем процент охвата. Например, охват 10% — это только первые 10% корзин control (в нашем примере 0–49 из 0–499 получат вариант control).

## Flagr на практике

- Допустим, мы хотим выкатить новую кнопку пользователям в США, но не знаем, какой цвет сработает лучше. Цвета `green/blue/pink` — это три варианта флага.
![](images/flagr_running_example_1.png)
![](images/flagr_running_example_4.png)

- Возможно, сначала стоит показать флаг небольшой группе — например, пользователям из Калифорнии. Вот они и есть сегмент.
![](images/flagr_running_example_2.png)

- Позже выяснилось, что в CA заходит зелёная кнопка, в NY — розовая, а в DC — синяя. Значит, у нас три сегмента, и каждый задан условием `state == ?`. Сегмент можно описать и несколькими условиями, например `state == NY AND Age >= 21`.
![](images/flagr_running_example_3.png)
![](images/flagr_running_example_5.png)

- Для A/B-теста по этому флагу можно взять распределение `50%/50%` между `green/blue` и проверить его лишь на `20%` (охват) пользователей сегмента `CA`. Потом охват можно поднять до `100%`, чтобы каждый пользователь в `CA` с вероятностью `50%` получал либо green, либо blue. А если нужно выкатить `100%` green на всех пользователей — просто поставьте распределение `100%/0%` для green/blue и охват `100%`.

## Архитектура Flagr

В Flagr три компонента: Flagr Evaluator, Flagr Manager и Flagr Metrics.

- **Flagr Evaluator** — оценивает входящие запросы.
- **Flagr Manager** — CRUD-шлюз. Все изменения флагов проходят здесь.
- **Flagr Metrics** — конвейер данных для сбора результатов оценки. В качестве бэкендов конвейера поддерживаются Kafka, Kinesis и Google Cloud Pubsub (выбирается через `FLAGR_RECORDER_TYPE`).
