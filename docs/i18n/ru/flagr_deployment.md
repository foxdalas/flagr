# Развёртывание и эксплуатация

Flagr поставляется как единый самодостаточный бинарник (и Docker-образ), который отдаёт API и веб-UI с одного порта (`18000`). Эта страница охватывает решения, которые важны при реальной эксплуатации: где хранятся данные, как масштабироваться и как закрыть доступ. У каждой упомянутой здесь настройки есть строка в [Конфигурации сервера](flagr_env).

## Выбор базы данных

`FLAGR_DB_DBDRIVER` + `FLAGR_DB_DBCONNECTIONSTR` выбирают хранилище:

| Драйвер | Когда использовать | Примечания |
|---|---|---|
| `sqlite3` (по умолчанию) | Локальная разработка, один узел, демо | Пишет в **файл** — только один узел (см. раздел о персистентности ниже) |
| `mysql` | Продакшен, несколько реплик | Общая БД позволяет горизонтальное масштабирование |
| `postgres` | Продакшен, несколько реплик | То же, что и MySQL |
| `json_file` / `json_http` | Узлы только для **оценки** | Загружает флаги из JSON-снимка; автоматически включает `FLAGR_EVAL_ONLY_MODE` (без API управления). См. [JSON-источник флагов](flagr_json_flag_spec) |

```bash
export FLAGR_DB_DBDRIVER=mysql
export FLAGR_DB_DBCONNECTIONSTR="user:pass@tcp(mysql:3306)/flagr?parseTime=true"
```

## Персистентность и резервные копии

!> **База данных SQLite по умолчанию находится внутри контейнера и теряется при его замене.** Опубликованный образ к тому же содержит *демонстрационную* базу по пути `/data/demo_sqlite3.db`. Если запустить стоковый образ, не изменив это, при следующем развёртывании вы незаметно потеряете свои флаги.

Для всего, что выходит за рамки одноразового демо:

- **SQLite** — укажите в строке подключения путь на **смонтированном томе**, чтобы файл переживал перезапуски, и делайте резервную копию этого файла. SQLite подходит для одного узла, но его нельзя разделять между репликами.
- **MySQL / Postgres** — поднимите управляемую/реплицированную базу данных и делайте её резервные копии, как и для любого другого продакшен-хранилища. Это рекомендуемый путь для продакшена.

В любом случае вы также можете в любой момент снять снимок всей конфигурации флагов через `GET /api/v1/export/sqlite` (см. [API управления](flagr_management_api)).

## Горизонтальное масштабирование

Flagr **не хранит состояние**, кроме как в базе данных, поэтому масштабироваться можно запуском дополнительных реплик за балансировщиком нагрузки — все они указывают на **одну и ту же** MySQL/Postgres:

- Каждая реплика держит **кэш оценки в памяти** и обновляет его из БД каждые `FLAGR_EVALCACHE_REFRESHINTERVAL` (по умолчанию **3 с**). Оценки обслуживаются целиком из этого кэша, поэтому они быстры и не обращаются к БД на каждый запрос.
- Платой за это становится **задержка распространения**: изменение флага, сделанное на одной реплике, становится видимым на остальных не позже чем через один интервал обновления (~3 с). Учитывайте это в тестах и при развёртываниях.
- **SQLite не может обслуживать несколько реплик** — его файл локален для одного узла. Для масштабирования вширь используйте MySQL/Postgres.

Распространённая топология для высокой нагрузки: небольшое **управляющее** развёртывание (полный API, владеет записями) плюс крупный пул реплик **только для оценки** (`FLAGR_EVAL_ONLY_MODE=true` либо драйвер `json_http`), которые обслуживают только `/api/v1/evaluation*` и [OFREP](flagr_ofrep).

## Защита инстанса

По умолчанию у Flagr **нет аутентификации** — любой, кто может до него достучаться, может читать *и записывать* флаги. Прежде чем выставлять его наружу, включите один из механизмов аутентификации (подробности и переменные окружения — в [Конфигурации сервера](flagr_env)):

| Механизм | Подходит для |
|---|---|
| **Basic auth** (`FLAGR_BASIC_AUTH_ENABLED`) | Простой защиты через общие учётные данные |
| **JWT** (`FLAGR_JWT_AUTH_ENABLED`) | Токенов из ваших SSO/прокси через куки или заголовки; поддерживает `HS256`/`HS512`/`RS256` |
| **Header auth** (`FLAGR_HEADER_AUTH_ENABLED`) | Доверия вышестоящему прокси, который добавляет заголовок с идентификатором |

!> Белые списки аутентификации могут для удобства оставлять оценку (а иногда и `/api/v1/flags`) открытой. **Проверьте, что именно открывает выбранный вами белый список** — убедитесь, что записывающие эндпоинты действительно требуют учётных данных, например выполните `curl` для `POST /api/v1/flags` без токена и проверьте, что запрос отклонён.

**CORS:** настройки по умолчанию разрешительны — `FLAGR_CORS_ALLOWED_ORIGINS=*` вместе с `FLAGR_CORS_ALLOW_CREDENTIALS=true`. В продакшене ужесточите `ALLOWED_ORIGINS` до реальных источников вашего UI.

**Профилирование:** `pprof` включён по умолчанию по адресу `/debug/pprof/` — не оставляйте его публично доступным (см. [Мониторинг](flagr_monitoring)).

## За обратным прокси

Чтобы отдавать Flagr по вложенному пути (например, `https://tools.example.com/flagr`), задайте `FLAGR_WEB_PREFIX=/flagr`; Flagr срезает префикс, а UI соответственно строит свои URL для API. На прокси пробрасывайте стандартные заголовки `X-Forwarded-*` и убедитесь, что ваши CORS-источники совпадают с публичным именем хоста.

## Kubernetes

Запускайте образ как обычный Deployment без состояния, указывающий на общую MySQL/Postgres, и подключите [эндпоинт работоспособности](flagr_monitoring) к обеим пробам:

```yaml
livenessProbe:
  httpGet: { path: /api/v1/health, port: 18000 }
readinessProbe:
  httpGet: { path: /api/v1/health, port: 18000 }
```

Масштабируйте через `replicas:` и помните про задержку распространения кэша ~3 с. Для пула только для оценки задайте на этих подах `FLAGR_EVAL_ONLY_MODE=true` (или используйте источник `json_http`).

## Работоспособность и наблюдаемость

`GET /api/v1/health` возвращает `200 {"status":"OK"}`, как только сервер поднят — используйте его для проб балансировщика нагрузки и Kubernetes. О метриках, трассировке и оповещениях см. [Мониторинг и метрики](flagr_monitoring).
