# API оценки

Оценка — это то, как ваше приложение спрашивает у Flagr: *«для этой сущности какой вариант этого флага включён?»* Это обычный HTTP-запрос — SDK не требуется, хотя для нескольких языков есть SDK от сообщества.

!> Предпочитаете стандарт OpenFeature? Flagr также поддерживает [OFREP](flagr_ofrep) по адресу `/ofrep/v1/...`, так что можно использовать любой OpenFeature SDK.

## Одиночная оценка

`POST /api/v1/evaluation`

```sh
curl -X POST http://localhost:18000/api/v1/evaluation \
  -H 'Content-Type: application/json' \
  -d '{
    "entityID": "user-123",
    "entityType": "user",
    "entityContext": { "state": "CA", "age": 31 },
    "flagKey": "checkout-redesign",
    "enableDebug": true
  }'
```

### Поля запроса

| Поле | Тип | Описание |
|-------|------|-------------|
| `entityID` | string | Стабильный идентификатор сущности. **Определяет корзину охвата/распределения** — одна и та же `entityID` всегда получает один и тот же вариант. Если не указан, Flagr генерирует случайный (без закрепления). |
| `entityType` | string | Необязательная метка сущности (например, `user`, `device`). Записывается вместе с метриками. **Если у самого флага задан Entity Type, побеждает его значение — оно переопределяет то, что вы передаёте здесь.** |
| `entityContext` | object | Атрибуты, по которым проверяются [условия](flagr_operators) сегмента. Имена `property` в условиях разрешаются по ключам *внутри* `entityContext` — условие на `country` читает `entityContext.country`, а не поле верхнего уровня. |
| `flagKey` | string | Указать флаг по ключу… |
| `flagID` | integer | …или по числовому ID. И то и другое ведёт к одному флагу. |
| `enableDebug` | boolean | При `true` ответ включает `evalDebugLog` с объяснением, почему каждый сегмент совпал или нет. |

### Поля ответа

```json
{
  "flagID": 42,
  "flagKey": "checkout-redesign",
  "flagTags": ["checkout", "web"],
  "flagSnapshotID": 7,
  "segmentID": 100,
  "variantID": 200,
  "variantKey": "treatment",
  "variantAttachment": { "color": "#42b983" },
  "timestamp": "2026-06-22T12:00:00Z",
  "dataRecordsEnabled": false,
  "evalDebugLog": { "segmentDebugLogs": [], "msg": "" }
}
```

| Поле | Описание |
|-------|-------------|
| `variantKey` / `variantID` | Назначенный вариант. **Пусто / `0` означает, что варианта нет** — обрабатывайте это как значение по умолчанию. |
| `variantAttachment` | JSON-вложение варианта, для [динамической конфигурации](flagr_use_cases). |
| `segmentID` | Какой сегмент совпал (`0`, если ни один). |
| `flagTags` | Теги совпавшего флага (возвращаются обратно — полезно, когда вы выбирали флаги по тегу). |
| `flagSnapshotID` | Использованная ревизия флага — удобно для сопоставления со вкладкой [истории](flagr_overview). |
| `dataRecordsEnabled` | Была ли эта оценка записана в конвейер метрик. |
| `evalDebugLog` | Трассировка совпадений по сегментам. Заполняется только при **обоих** условиях: `enableDebug: true` в запросе **и** `FLAGR_EVAL_DEBUG_ENABLED=true` на сервере (значение по умолчанию). |

Ваш код ветвится по `variantKey`:

```js
const res = await fetch(`${FLAGR}/api/v1/evaluation`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    entityID: userId,
    flagKey: "checkout-redesign",
    entityContext: { state },
  }),
}).then((r) => r.json());

if (res.variantKey === "treatment") {
  // new experience
} else {
  // control / default — also the path when no variant matched
}
```

!> **Всегда обрабатывайте случай «варианта нет».** Выключенный флаг, сущность, не подошедшая ни под один сегмент, или сущность за пределами охвата — все они возвращают пустой `variantKey`. Считайте это безопасным значением по умолчанию. См. [Как работает оценка](flagr_evaluation).

## Массовая (bulk) оценка

Оцените много сущностей и/или много флагов за один запрос.

`POST /api/v1/evaluation/batch`

```json
{
  "entities": [
    { "entityID": "u1", "entityContext": { "state": "CA" } },
    { "entityID": "u2", "entityContext": { "state": "NY" } }
  ],
  "flagIDs": [42, 43],
  "enableDebug": false
}
```

Ответ — `{ "evaluationResults": [ ... ] }`, по одному `evalResult` на каждую пару (сущность × выбранный флаг).

!> `FLAGR_EVAL_BATCH_SIZE` ограничивает `entities × flags`, чтобы защитить сервер (по умолчанию `0` = без ограничений). Запросы сверх лимита отклоняются.

### GET-вариант batch (удобно для lambda)

`GET /api/v1/evaluation/batch` принимает выборку через query-параметры — для вызывающих, которые умеют только GET (CDN, edge-среды выполнения). Он поддерживает `ETag` / `If-None-Match`, поэтому при неизменной конфигурации флагов возвращается `304 Not Modified` — дёшево опрашивать.

Имена query-параметров отличаются от полей тела POST и все могут повторяться (мультизначные):

| Query-параметр | Соответствует |
|-------------|---------|
| `entityId` | идентификатор сущности (одна сущность на запрос) |
| `flagId` / `flagKey` | выбор флагов по ID или ключу |
| `flagTag` | выбор флагов по тегу (повторяйте для нескольких) |
| `flagTagQuery` | `ANY` (по умолчанию) или `ALL` — как объединяются значения `flagTag` |

Два ограничения GET-варианта по сравнению с POST batch:

- **Одна сущность на запрос** (`entityId`), и её **`entityType` всегда `user`** — задать произвольный тип сущности в GET-форме нельзя.
- Контекст сущности передаётся как дополнительные query-параметры (каждый становится ключом контекста).

## Выбор флагов для оценки

Выберите один из вариантов:

- **`flagKey`** / **`flagKeys`** — самый читаемый; стабилен между окружениями.
- **`flagID`** / **`flagIDs`** — числовой; совпадает с URL флага в UI.
- **`flagTags`** с **`flagTagsOperator`** — оценить каждый флаг с этими тегами. `ANY` (по умолчанию) = есть хотя бы один из тегов; `ALL` = есть все. Идеально для сценария «оценить все флаги для этой поверхности».

!> **Выбор по тегам работает только в batch.** Одиночный `POST /api/v1/evaluation` разрешает флаг только по `flagID`/`flagKey` — он игнорирует `flagTags`. Чтобы оценивать по тегу, используйте batch-эндпоинт.

## Детерминизм: передавайте стабильный entityID

Охват и распределение детерминированы по `entityID`: одна и та же сущность всегда попадает в одну и ту же корзину, поэтому пользователь сохраняет один и тот же вариант между запросами и перезапусками. Передавайте **стабильный** идентификатор (ID пользователя, ID аккаунта, ID устройства) — а не случайное значение на каждый запрос, иначе закрепление варианта теряется. См. [Обзор → Охват](flagr_overview).

## Узлы только для оценки

Для высоконагруженных конфигураций запускайте отдельные инстансы оценки с `FLAGR_EVAL_ONLY_MODE=true`: они обслуживают `/api/v1/evaluation*` (и [OFREP](flagr_ofrep)) из кэша в памяти и отключают CRUD/управляющий API. Дополните их отдельным управляющим инстансом, которому принадлежат записи.

## Клиентские библиотеки

SDK от сообщества (Go, JavaScript, Python, Ruby) оборачивают эти эндпоинты — см. README проекта. Для OpenFeature направьте любой OpenFeature SDK на эндпоинт [OFREP](flagr_ofrep) у Flagr; специфичный для Flagr клиент не нужен.
