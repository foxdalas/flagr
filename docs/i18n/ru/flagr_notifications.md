# Уведомления

В Flagr встроена система уведомлений, которая позволяет в реальном времени отслеживать изменения операционных ресурсов. Flagr можно настроить так, чтобы он отправлял HTTP-вебхуки методом `POST` всякий раз, когда флаг создаётся, изменяется, удаляется или восстанавливается.

## Отслеживаемые операции

Flagr отправляет уведомление на каждое изменение флага **и любой из его частей** — сегментов, вариантов, условий, распределений и тегов. В каждом уведомлении есть два поля, которые вместе точно описывают, что произошло: **`operation`** и **`component_type`**.

| `operation` | Значение |
|-----------|-------------|
| `create` | Создан флаг или одна из его частей |
| `update` | Изменён флаг или одна из его частей |
| `delete` | Удалён флаг (soft-delete) или одна из его частей |
| `restore` | Восстановлен помеченный как удалённый флаг |

| `component_type` | Что изменилось |
|------------------|-----------------------|
| `flag` | Сам флаг — метаданные, состояние включён/выключен |
| `segment` · `variant` · `constraint` · `distribution` · `tag` | Конкретный дочерний элемент флага |

Эти два поля комбинируются, поэтому уведомление точно указывает на вложенное изменение:

- Добавление варианта → `operation: create`, `component_type: variant`
- Изменение охвата сегмента → `operation: update`, `component_type: segment`
- Удаление тега → `operation: delete`, `component_type: tag`
- Включение/выключение флага → `operation: update`, `component_type: flag`

Каждое уведомление также содержит `flag_id` / `flag_key` родительского флага, так что получатель может группировать дочерние изменения по их флагу.

## Настройка

Чтобы включить уведомления, задайте следующие переменные окружения:

- `FLAGR_NOTIFICATION_WEBHOOK_ENABLED=true` (по умолчанию: `false`) — включает уведомления через вебхуки.
- `FLAGR_NOTIFICATION_WEBHOOK_URL=https://api.your-org.com/webhooks/flagr` — HTTP-эндпоинт назначения для POST-запросов.
- `FLAGR_NOTIFICATION_WEBHOOK_HEADERS=Authorization: Bearer secret-token, X-Custom-Header: value` — (необязательно) произвольные HTTP-заголовки через запятую; обычно используются, чтобы защитить приёмник вебхуков API-токеном.
- `FLAGR_NOTIFICATION_TIMEOUT=10s` (по умолчанию: `10s`) — общий таймаут на доставку одного уведомления, **включая все повторы** (а не только первичное соединение).
- `FLAGR_NOTIFICATION_DETAILED_DIFF_ENABLED=true` (по умолчанию: `false`) — когда включено, Flagr добавляет в тело уведомления подробный наглядный JSON-diff изменённого флага.
- `FLAGR_NOTIFICATION_MAX_RETRIES=3` (по умолчанию: `3`) — максимальное число повторных попыток при временных сбоях (сетевые ошибки и ошибки транспорта, а также ответы 5xx). Значение `0` отключает повторы.
- `FLAGR_NOTIFICATION_RETRY_BASE=1s` (по умолчанию: `1s`) — базовая задержка для экспоненциального бэкоффа между повторами.
- `FLAGR_NOTIFICATION_RETRY_MAX=10s` (по умолчанию: `10s`) — максимальная задержка между повторами.

### Параллелизм и наблюдаемость

- Уведомления отправляются асинхронно с фиксированным ограничением параллелизма в 100 (не настраивается), чтобы не исчерпать ресурсы под нагрузкой.
- Когда включён statsd, отправляется метрика `notification.sent` с тегами `provider`, `operation` и `status` (`success`/`failure`).

### Важные замечания

- **Асинхронная доставка**: уведомления отправляются в фоновых горутинах. Сбои логируются, но **не влияют на ответ API**.
- **Проверка при старте**: Flagr проверяет конфигурацию уведомлений при запуске и пишет предупреждение, если вебхуки включены, но `FLAGR_NOTIFICATION_WEBHOOK_URL` не задан.
- **Тихий фолбэк**: если вебхуки включены, а URL отсутствует, уведомления молча отбрасываются. При старте пишется предупреждение, чтобы помочь диагностировать ошибку конфигурации.

## Формат тела вебхука

Целевой эндпоинт получает структурированное тело запроса в формате JSON:

```json
{
  "operation": "update",
  "flag_id": 123,
  "flag_key": "my-feature-flag",
  "component_type": "segment",
  "component_id": 7,
  "component_key": "power-users",
  "pre_value": "...",
  "post_value": "...",
  "diff": "--- Previous\n+++ Current\n@@ ...",
  "user": "admin@example.com",
  "timestamp": "2026-04-26T18:51:03Z"
}
```

| Поле | Тип | Описание |
|-------|------|-------------|
| `operation` | string | `create`, `update`, `delete` или `restore` |
| `flag_id` | uint | ID родительского флага в базе данных |
| `flag_key` | string | Уникальный ключ родительского флага |
| `component_type` | string | Что именно изменилось во флаге: `flag`, `segment`, `variant`, `constraint`, `distribution` или `tag` |
| `component_id` | uint | ID изменённого компонента в базе данных |
| `component_key` | string | Ключ или имя изменённого компонента (например, ключ варианта, значение тега) |
| `pre_value` | string | JSON-снимок предыдущего состояния флага (только если `FLAGR_NOTIFICATION_DETAILED_DIFF_ENABLED=true`)
| `post_value` | string | JSON-снимок текущего состояния флага (только если `FLAGR_NOTIFICATION_DETAILED_DIFF_ENABLED=true`)
| `diff` | string | Unified diff между предыдущим и текущим состоянием (только если `FLAGR_NOTIFICATION_DETAILED_DIFF_ENABLED=true`)
| `user` | string | Кто из пользователей внёс изменение |
| `timestamp` | string | Метка времени изменения в UTC в формате RFC 3339 |
