# Как работает оценка

Когда вы оцениваете флаг для **сущности** (пользователя, устройства или запроса — чего угодно с идентификатором и необязательным контекстом), Flagr проходит по одному и тому же маршруту и возвращает **ровно один вариант** — или ни одного. Если понимать этот маршрут, почти все вопросы про сегменты, охват и распределение отпадают сами собой.

## Маршрут оценки

1. **Флаг включён?** Выключенный флаг не возвращает ни одного варианта — на этом оценка заканчивается.
2. **Идём по сегментам сверху вниз.** Сегменты упорядочены, и **побеждает первый подходящий**. Как только сегмент совпал, Flagr дальше вниз уже не смотрит.
3. **Сущность проходит по условиям сегмента?** Все условия внутри сегмента объединяются по `AND`. Сегмент **без условий подходит всем**.
4. **Сущность попадает в охват?** У подходящего сегмента есть **охват %** — доля подходящих сущностей, которые реально в него включаются. Охват детерминирован для каждой сущности (одна и та же сущность всегда попадает одинаково), поэтому охват 20% — это всегда одни и те же 20%.
5. **Выбираем вариант по распределению.** Для включённой сущности **распределение** сегмента решает, какой вариант она получит (например, 50% `on` / 50% `off`).

!> Порядок важен: ставьте самые узкие сегменты выше. Если широкий сегмент (например, «все») окажется над узким, первым совпадёт широкий, и до узкого дело не дойдёт. Чтобы поменять порядок, перетащите сегмент за маркер.

## Охват и распределение

Это два разных шага, и их часто путают:

- **Охват %** отвечает на вопрос *«попадает ли сущность в сегмент вообще?»* — он решает, включать её или нет.
- **Распределение** отвечает на вопрос *«какой вариант получит уже включённая сущность?»* — оно делит включённый трафик между вариантами.

Поэтому сегмент с **охватом 100%** и **распределением 50/50** даёт вариант всем в сегменте — половине `on`, половине `off`. А сегмент с **охватом 20%** включает лишь 20% подходящих сущностей; остальные 80% **не получают варианта**.

!> **Совпавший сегмент завершает оценку.** «Побеждает первый подходящий» означает первый сегмент, чьи *условия* совпали. Как только это произошло, Flagr **не** смотрит на сегменты ниже — даже если сущность затем не попала в охват и осталась без варианта. Переход к следующему сегменту происходит только тогда, когда условия *не* совпадают. Поэтому узкий сегмент с охватом 20%, поставленный над «ловушкой для всех», оставит 80% своих подходящих сущностей без варианта, а не передаст их вниз в «ловушку».

## Почему это детерминированно

Охват и распределение не бросают кости — они хешируют сущность в один из **1000 бакетов**:

```
bucket = crc32( flagID + entityID ) mod 1000
```

Хеш солится **идентификатором флага**, и из этого следуют два свойства:

- **Стабильность для каждой сущности и каждого флага.** Одна и та же `entityID` всегда попадает в один и тот же бакет для конкретного флага, поэтому пользователь сохраняет один и тот же вариант между вызовами (закрепление), а охват 20% всегда включает *одни и те же* 20%.
- **Независимость между флагами.** Поскольку соль — это идентификатор флага, один и тот же пользователь попадает в *разные* бакеты в каждом флаге: участие в экспериментальной группе одного эксперимента ничего не говорит о другом. Никакой корреляции между флагами.

1000 бакетов делятся между вариантами по процентам **распределения** (при делении 50/50 одному варианту достаются бакеты 0–499, другому — 500–999). Затем **охват %** применяется *внутри* полосы бакетов нужного варианта: при охвате 100% включается каждый бакет полосы, при охвате 20% — только первая пятая её часть. Вот почему охват и распределение — два разных шага: бакет сначала выбирает вариант, а охват затем решает, включён ли этот бакет вообще.

Чтобы получить незакреплённый разовый результат, отправьте пустой `entityID` — Flagr сгенерирует для этого единственного вызова случайный (он всё равно проходит через тот же хеш, так что результат внутренне согласован, просто не повторяем).

## Когда вариант не вернётся?

В нескольких случаях вариант не возвращается. Обычно это ошибки в настройке, и о последних двух страница флага теперь предупреждает:

- **Флаг выключен.**
- **Ни один сегмент не подошёл** — контекст сущности не прошёл ни по одному набору условий, а сегмента-«ловушки» для всех нет.
- **Сущность подошла под сегмент, но не попала в его охват %** — например, охват равен 0%, и тогда не включается никто.
- **У подходящего сегмента нет распределения** — выдавать просто нечего.

!> Последние две ситуации страница флага подсветит сама: сегмент с охватом 0% или без распределения показывает предупреждение, чтобы эксперимент случайно не «запустился» без единого участника.

## Проверьте на реальной сущности

Необязательно держать всё это в голове:

- **Консоль отладки** (на странице флага) прогоняет тестовую сущность через флаг и показывает, какой сегмент совпал и какой вариант она получила. Режим только для чтения — на боевой трафик это не влияет.
- **Поток оценки** (вкладка на странице флага) наглядно показывает весь этот маршрут для тестовой сущности.

!> Строковые значения в условиях берите в кавычки — `"CA"`, а не `CA`. Без кавычек значение считается именем переменной и не совпадёт никогда — причём без всякого предупреждения.
